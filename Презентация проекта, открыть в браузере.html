Ладья — децентрализованная система связи без центрального сервера

P2P / Mesh-мессенджер для обмена сообщениями, файлами и real-time-коммуникации внутри локальной сети
Ключевая идея
Мы создаём систему, в которой каждое устройство становится полноценным сетевым узлом: обнаруживает соседей, устанавливает прямые соединения и при необходимости ретранслирует трафик для других участников.
  • Связь не зависит от облака и внешнего сервера
  • Работает внутри локальной сети и mesh-топологии
  • Подходит для сценариев перегрузки или недоступности интернета
Основной акцент проекта — надёжный протокол обмена, multihop-маршрутизация и демонстрация работы в реальном времени.

Проблема

Почему централизованные мессенджеры не всегда подходят

Если сервер или интернет недоступен, коммуникация останавливается — даже когда устройства физически находятся рядом
Современные системы связи почти всегда опираются на внешнюю инфраструктуру: облако, мобильный интернет, публичные серверы и сторонние платформы.
  • Перегрузка сети на массовых мероприятиях
  • Ограниченный доступ к интернету на закрытых площадках
  • Единая точка отказа в виде сервера
  • Риск вывода данных за пределы локального контура

Надёжность

Что делает сеть устойчивой

Мы отдельно прорабатываем корректную доставку, повторные попытки и поведение системы при разрывах
Механизмы устойчивости
  • ✓ ACK-подтверждения доставки
  • ✓ Retry-повторы при временной недоступности
  • ✓ Очереди сообщений и контроль перегрузки relay-узлов
  • ✓ Дедупликация пакетов и защита от циклов
  • ✓ Корректная реакция на отключения и смену маршрута
Что это даёт на практике
  • → Сообщения не теряются при кратковременных сбоях
  • → Сеть не «зацикливает» пакеты при relay-передаче
  • → Интерфейс может показывать понятный статус доставки
  • → Архитектура готова к тестам на отказ и нестабильную сеть
Передача файлов

Файлы передаются как протокол, а не как «чёрный ящик»

Это важно, потому что файл не должен блокировать чат и должен восстанавливаться после обрыва
Chunk-based file transfer
File
→
Chunk 1
→
Chunk 2
→
Chunk 3
Идея: файл дробится на части, каждая часть подтверждается, а при обрыве передача продолжается не с нуля, а с последнего подтверждённого блока.
Ключевые свойства
  • ✓ Хэш и контроль целостности
  • ✓ Подтверждение получения чанков
  • ✓ Возможность возобновления передачи
  • ✓ Ограничение нагрузки, чтобы не «убивать» канал сообщений
  • ✓ Понятный статус прогресса для пользователя
