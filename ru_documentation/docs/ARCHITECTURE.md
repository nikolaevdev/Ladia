# Архитектура Ladya

## 1. Архитектурный обзор

Ladya — это Android-мессенджер децентрализованного типа, построенный вокруг **одного локального узла**, который запускается на каждом устройстве.
Каждый экземпляр приложения выступает как самостоятельный сетевой узел и умеет:

- публиковать себя в сети;
- обнаруживать другие узлы;
- устанавливать прямые сессии;
- ретранслировать пакеты для других peer-узлов;
- передавать файлы;
- участвовать в групповой доставке;
- выполнять аудио- и видеозвонки;
- поддерживать локальную идентичность и trust-state;
- сохранять состояние чатов в SQLite.

С точки зрения архитектуры текущая реализация представляет собой **монолитную, но послойную Android MVP-систему**, в которой один крупный repository координирует практически весь жизненный цикл коммуникаций.

## 2. Основные архитектурные слои

### UI layer

Расположен в:
- `ui/screens/`
- `ui/components/`
- `ui/models/`
- `navigation/`

Зона ответственности:
- отрисовка Compose-интерфейса;
- обработка пользовательского ввода;
- отображение состояния соединения, чатов, звонков, логов, профиля и настроек;
- передача пользовательских действий в repository.

Важная особенность:
UI построен по state-driven подходу. Большинство экранов работают с неизменяемыми моделями состояния, например:
- `ConnectionUiState`
- `ChatUiState`
- `CallUiState`
- `ContactsUiState`
- `SettingsUiState`
- `ProfileUiState`

### Application / service layer

Расположен в:
- `MainActivity.kt`
- `service/LadyaForegroundService.kt`
- `notify/AppNotificationManager.kt`

Зона ответственности:
- bootstrap приложения;
- awareness foreground/background-режима;
- уведомления;
- поддержание жизни узла во время длительных сетевых операций.

### Core coordination layer

Расположен в:
- `network/LadyaNodeRepository.kt`

Это главный оркестратор проекта.
На текущем этапе он объединяет:
- discovery;
- управление прямыми соединениями;
- relay-логику;
- маршрутизацию;
- профили;
- trust;
- сохранение чатов;
- file transfer;
- group logic;
- координацию с аудио/видео звонками;
- системные логи;
- часть diagnostics.

Практически всё сетевое поведение приложения проходит через этот класс.

### Protocol layer

Расположен в:
- `network/protocol/PacketEnvelope.kt`
- `network/protocol/PacketFactory.kt`
- `network/protocol/PacketParser.kt`
- `network/protocol/ProtocolConstants.kt`
- `network/protocol/TransferManifest.kt`

Зона ответственности:
- формальное описание transport envelope;
- типы пакетов;
- сериализация и десериализация;
- manifest-схема передачи файлов;
- TTL, retry, route- и packet-константы.

### Media transport layer

Расположен в:
- `call/AudioCallManager.kt`
- `call/VideoCallManager.kt`
- частично в `sound/AppSoundManager.kt`

Зона ответственности:
- real-time транспорт медиа;
- захват микрофона и камеры;
- аудио/видео packet loop;
- bitrate/profile adaptation;
- восстановление удалённых кадров;
- метрики и обработка ошибок звонков.

### Security / trust groundwork layer

Расположен в:
- `security/DeviceIdentityManager.kt`
- `security/StegoTrustCapsuleManager.kt`

Зона ответственности:
- идентичность устройства;
- подпись и проверка handshake payload;
- fingerprint и short authentication string;
- trust capsule для изображений;
- groundwork для Stage 5.

Важно:
это ещё не полноценный E2EE-слой, а задел под него.

### Persistence layer

Расположен в:
- `network/LadyaDatabaseHelper.kt`

Зона ответственности:
- хранение чатов;
- сохранение сообщений;
- потоки и контакты;
- часть метаданных профилей/состояния;
- восстановление данных для UI после перезапуска.

## 3. Runtime-архитектура

Текущий runtime использует несколько каналов связи одновременно.

### 3.1 Discovery channel

Используется для публикации и обнаружения узлов в одной локальной сети.
Ключевая идея:
узел рекламирует собственный сервис, а другие узлы его находят через NSD.

Назначение:
- первичное обнаружение peer-узлов;
- отображение найденных устройств на экране соединения;
- запуск прямого подключения.

### 3.2 Main direct session channel

Основной канал прямого общения между двумя узлами.
Как правило, это TCP-сессия, по которой передаются:
- сообщения;
- handshake;
- control packets;
- часть служебных packet envelope.

Именно этот канал является основой для P2P text chat и части control-plane логики.

### 3.3 File channel

Отдельный канал file transfer используется для передачи больших payload.
Обычно через него идут:
- transfer manifest;
- chunks;
- completion/status события.

Такое разделение позволяет не перегружать основной messaging channel.

### 3.4 Audio call channel

Аудиозвонки используют UDP-передачу и свой жизненный цикл.
Слой audio call отвечает за:
- захват микрофона;
- packetization;
- отправку;
- приём;
- playback;
- метрики и ошибки.

### 3.5 Video call channel

Видео также идёт по UDP и отделено от текстового/файлового транспорта.
Поддерживаются:
- захват кадров камеры;
- облегчённая упаковка кадров;
- профиль качества;
- реконструкция удалённого видеопотока.

### 3.6 Relay channel

Relay-канал используется для multihop-передачи через промежуточные узлы.
По нему проходят:
- relay packet envelope;
- file relay packets;
- route retry / deduplication логика;
- mesh diagnostics.

Ключевой принцип:
relay-узел пересылает пакет дальше, не являясь конечным участником пользовательского диалога.

## 4. Зона ответственности центрального repository

`LadyaNodeRepository` — это центр приложения. Сейчас он отвечает за:

- текущее состояние узла;
- найденных peer-узлов;
- активные соединения;
- логику соединения и reconnection;
- обработку пакетов;
- file transfer;
- trust-состояния;
- списки чатов и сообщений;
- работу с БД;
- групповую доставку;
- взаимодействие с audio/video manager;
- diagnostics counters;
- выдачу state для Compose UI.

### Архитектурное следствие

Это даёт быстрый темп разработки и упрощает координацию этапов, но создаёт ряд рисков:
- класс становится слишком большим;
- изменения в одной области могут затрагивать другую;
- сложнее тестировать отдельные функции;
- труднее безопасно внедрять новый security-layer.

Поэтому Stage 5 и последующие этапы желательно делать через **новые модульные компоненты**, а не через ещё больший рост repository.

## 5. Модель пакета

### 5.1 Envelope

`PacketEnvelope` — это центральная транспортная модель проекта.
Именно через неё обеспечиваются:
- тип пакета;
- sender/target metadata;
- packet id;
- route-related поля;
- TTL;
- payload;
- признаки служебной доставки.

Envelope уже используется во всей mesh-пайплайне, поэтому ломать его нельзя.
Новые security-функции должны встраиваться поверх существующего envelope.

### 5.2 Почему envelope критически важен

Если изменить структуру envelope слишком радикально, можно сломать:
- существующий packet parser;
- relay pipeline;
- diagnostics;
- file transfer;
- group delivery;
- маршрутизацию и deduplication.

Поэтому Stage 5 должен:
- сохранять envelope-совместимость;
- шифровать payload, а не транспортную оболочку;
- добавлять security-marker поля аккуратно и обратно совместимо.

## 6. Handshake и trust-модель

### Текущий handshake flow

Сейчас в проекте уже есть handshake уровня идентичности:
- `HELLO`
- `HELLO_ACK`
- `HELLO_CONFIRM`

В ходе него стороны:
- обмениваются идентичностью;
- подписывают данные;
- сравнивают и проверяют подписи;
- формируют fingerprint / SAS;
- фиксируют trust-state.

Это важный задел под Stage 5, потому что на его основе можно построить нормальную cryptographic binding-модель.

### Текущие trust-состояния peer-узла

Поддерживаются состояния:
- `Unverified`
- `Verified`
- `Suspicious`
- `Blocked`

### Уже реализованное trust-aware поведение

Система уже умеет:
- предупреждать о подозрительной смене ключа;
- исключать blocked-узлы из доверенных маршрутов;
- предпочитать verified relay paths;
- показывать fingerprint и trust-информацию в UI.

## 7. Архитектура сообщений

### Путь прямого сообщения

Типичный direct path:
1. пользователь вводит сообщение в UI;
2. UI вызывает action repository;
3. repository формирует packet envelope;
4. packet сериализуется и отправляется в прямую TCP-сессию;
5. remote repository принимает пакет, разбирает его и обновляет локальное состояние/БД/UI.

### Путь relay-сообщения

Типичный relay path:
1. прямой маршрут недоступен;
2. repository подбирает relay path;
3. пакет получает route metadata и отправляется relay-узлу;
4. промежуточный узел выполняет deduplication / TTL check / forward;
5. конечный получатель принимает пакет и обрабатывает его как обычное входящее сообщение.

## 8. Архитектура передачи файлов

Передача файлов идёт отдельным потоком и обычно включает:
- manifest с описанием файла;
- подтверждение готовности;
- отправку chunks;
- статус завершения.

В mesh-сценарии добавляется relay-delivery через промежуточные узлы.
На этом уровне уже есть важные элементы:
- логика доставки частей;
- progress tracking;
- сохранение вложений;
- сохранение по запросу пользователя;
- поддержка групповой доставки файлов.

Stage 5 должен не переписывать этот pipeline, а обернуть файл в **secure container**, после чего текущий transport сможет доставлять уже шифрованный blob.

## 9. Архитектура звонков

### Audio

Аудиозвонки реализованы через отдельный UDP-поток и менеджер звонка.
Ключевые задачи:
- чтение из микрофона;
- упаковка аудиопакетов;
- отправка;
- буферизация и воспроизведение на приёмной стороне;
- mute/speaker control;
- статистика звонка.

### Video

Видео строится схожим образом:
- захват кадров камеры;
- лёгкая пакетизация;
- передача по UDP;
- адаптация качества;
- реконструкция удалённых кадров.

### Важная оговорка по scope

Текущие звонки уже работают как MVP-связь реального времени, но ещё не являются полноценным production-grade media stack. Для Stage 5 важнее сначала защитить сообщения и контейнеры, чем пытаться сразу шифровать весь media-plane.

## 10. Групповая архитектура

Группы добавляют второй важный уровень сложности после mesh.
Система уже поддерживает:
- создание группы;
- приглашения;
- membership handling;
- owner/admin-права;
- изменение имени группы;
- исключение участников;
- отдельный экран группового чата;
- групповые текстовые сообщения;
- групповой file relay;
- групповую доставку голосовых сообщений.

Архитектурно группа сейчас представляет собой сущность поверх уже существующей direct/relay-сети.
То есть group-delivery не создаёт новый сетевой стек, а использует уже существующие transport и routing-пути.

Это полезно для Stage 5:
secure containers и E2EE можно внедрять как общий слой, который затем будет применяться и к direct-, и к relay-, и к group-обмену.

## 11. Архитектура persistence

SQLite используется как локальный источник состояния.
Сохраняются:
- чаты;
- сообщения;
- часть профилей и потоков;
- метаданные файлов;
- восстановимое состояние для UI.

Преимущества:
- приложение сохраняет историю после перезапуска;
- UI быстрее поднимается из локального состояния;
- легче показывать список потоков и сообщений.

Ограничение:
с точки зрения Stage 5 нужно отдельно решить, в каком виде хранить secure payload локально — как plaintext после успешного decrypt, как encrypted blob или смешанную модель.

## 12. Архитектура диагностики

Диагностика уже является важной частью приложения.
Система отслеживает:
- состояние соединения;
- найденных соседей;
- маршруты;
- relay counters;
- file counters;
- часть trust/topology информации.

Это полезно не только для отладки, но и для демонстрации на защите проекта.

Для Stage 5 сюда логично добавить:
- `encryptedPacketsSent`
- `encryptedPacketsReceived`
- `decryptionErrors`
- `stegoPacketsDetected`
- при необходимости дополнительные security counters.

## 13. Стратегия архитектурной интеграции Stage 5

Главный принцип Stage 5:
**не переписывать mesh, а надстраивать безопасность сверху**.

Это означает:
- не ломать `PacketEnvelope`;
- не менять радикально relay pipeline;
- не переписывать routing;
- использовать envelope как транспортную оболочку;
- шифровать payload и typed container внутри него.

### Рекомендуемые новые модули Stage 5

- `security/IdentityKeyManager.kt`
- `security/SessionKeyManager.kt`
- `security/SecurePayloadCipher.kt`
- `security/SecureContainer.kt`
- `security/SecureContainerCodec.kt`
- `security/StegoTransportManager.kt`
- `security/FingerprintQrCodec.kt`

Такие модули позволят локализовать security-логику и не размывать ещё сильнее `LadyaNodeRepository`.

## 14. Основные сильные стороны архитектуры

- уже есть реальная работающая P2P/mesh-коммуникация;
- UI, сеть, звонки, trust и persistence уже связаны в единую систему;
- есть diagnostics и topology visibility;
- реализованы группы, файлы, голос, видео;
- есть задел по trust и идентичности;
- security-слой можно реально встроить без полного переписывания.

## 15. Основные слабые стороны архитектуры

- repository слишком крупный;
- много логики сосредоточено в одном классе;
- уровень модульной изоляции пока недостаточен;
- автоматизированных тестов мало;
- security-layer ещё не завершён;
- часть протокольной логики тесно связана с runtime-состоянием.

## 16. Рекомендуемое следующее направление рефакторинга

После безопасного завершения Stage 5 логично двигаться в сторону:
- выделения connection/session manager;
- выделения relay/mesh coordinator;
- выделения message/file pipeline;
- отделения persistence adapter от repository;
- отделения trust/security orchestration в самостоятельный слой.

Но до этого шага важнее сохранить работоспособность текущих Stages 1–4.8.5 и аккуратно внедрить security-подсистему.
