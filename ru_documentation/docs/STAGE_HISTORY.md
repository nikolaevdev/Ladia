# История этапов

Этот документ кратко описывает эволюцию реализации, видимую в репозитории и в stage-specific markdown-файлах.

## Stage 1 — начальный Android P2P MVP

Главный результат:
- первая основа децентрализованного Android-мессенджера без центрального сервера.

Ключевые добавления:
- bootstrap локального узла;
- обнаружение peer-узлов в локальной сети;
- установление прямого соединения;
- текстовый чат;
- базовые контакты и потоки;
- основа локального профиля.

## Stage 2 — усиление протокола и профилей

Главный результат:
- repository стал более структурирован вокруг идентичности устройства и обмена профилями.

Ключевые добавления и изменения в рамках подэтапов Stage 2:
- улучшение поведения прямой сессии;
- groundwork вокруг keypair;
- распространение профиля;
- переход к SQL-based persistence чатов;
- постепенные правки UX и протокола.

## Stage 4 — mesh, relay, recovery, groups

Stage 4 — важнейшее сетевое расширение репозитория.
Именно оно превращает прямой P2P-мессенджер в **децентрализованную систему связи с поддержкой mesh**.

### Stage 4.1 — основа mesh recovery
Главные добавления:
- self-healing восстановление после потери активного peer-узла;
- сохранение listening/discovery mode;
- reconnect attempts;
- fallback на резервного соседа.

Почему это было важно:
узел перестал логически «умирать» при исчезновении одного прямого соседа.

### Stage 4.2 — mesh text relay
Главные добавления:
- отдельный relay listener на порту 1907;
- relay packet types;
- route labels в UI;
- deduplication relay-пакетов;
- pending relay queue;
- retry после route discovery.

Почему это было важно:
текстовый трафик стал проходить через другой peer-узел.

### Stage 4.3 — mesh file relay + diagnostics
Главные добавления:
- поддержка relay файлов через manifest/control/chunk/complete flow;
- visibility диагностики и топологии.

Почему это было важно:
проект вышел за рамки relay текста и начал передавать более сложный контент через mesh-слой.

### Stage 4.4 — trust-aware relay
Главные добавления:
- исключение blocked-маршрутов;
- предпочтение verified-маршрутов;
- предупреждения по unverified/suspicious relay path;
- diagnostics counters по trust-составу маршрутов.

Почему это было важно:
маршрутизация стала учитывать не только связность, но и доверие.

### Stage 4.5 — multi-neighbor reroute
Главные добавления:
- выбор лучшего relay-кандидата среди нескольких соседей;
- штраф для недавно отказавших relay-узлов;
- счётчики reroute и recovered-route;
- retry через альтернативный relay.

Почему это было важно:
mesh-слой стал более fault-tolerant.

### Stage 4.6 — основа групп + voice
Главные добавления:
- ранняя основа группового взаимодействия;
- groundwork по voice-взаимодействию в группах.

### Stage 4.7 — групповой чат поверх mesh
Главные добавления:
- создание группы;
- приглашения;
- групповые текстовые сообщения;
- отдельная модель потока `group:<id>`;
- group headers с membership;
- прямая или relay-доставка групповых сообщений в зависимости от доступного маршрута.

Почему это было важно:
система эволюционировала от попарного общения к многопользовательскому.

### Stage 4.8 — group file/voice/diagnostics
Главные добавления:
- MVP доставки файлов в группу через mesh;
- доставка голосовых сообщений в группу;
- групповые diagnostics counters;
- исправления вокруг group model definitions.

### Stage 4.8.1 — group audio / create UX
- доработка UX создания группы и сценариев, связанных с аудио.

### Stage 4.8.2 — отдельное окно группы
- выделенный маршрут/экран группового чата;
- отключение аудио/видеозвонков в групповом режиме;
- более чистое разделение routing и UX.

### Stage 4.8.3 — исправление group settings
- улучшение стабильности сценариев, связанных с настройками группы.

### Stage 4.8.4 — relay patch
- исправление compile-time defect в helper-логике relay forwarding.

### Stage 4.8.5 — group fixes / settings / save-on-demand
Главные добавления:
- исправлен сценарий выхода из группы;
- усилена логика открытия group settings;
- явные проверки owner/admin rights;
- групповые вложения продолжают приходить без confirm-flow;
- пользователь может сохранить копию group attachment / voice note по запросу из message bubble.

К концу Stage 4.8.5 проект уже имел:

## Stage 5 — security-слой в процессе

Stage 5 — следующий логичный шаг. Он должен добавить **конфиденциальность payload и более сильную привязку идентичности** поверх уже работающей mesh-сети.

Целевой scope:
- identity keys в Android Keystore;
- deriving session keys для каждого peer;
- AES-GCM secure message encryption;
- secure containers для файлов / голоса / изображений;
- стеганографический транспорт;
- encrypted blobs, совместимые с mesh;
- UX верификации fingerprint через QR.

## Важное правило преемственности

История проекта показывает жёсткое ограничение:
**новые этапы должны расширять предыдущие, а не ломать их**.

Для Stage 5 это означает:
- не переписывать mesh transport с нуля;
- не ломать `PacketEnvelope`;
- не заменять routing несовместимой логикой;
- добавлять security как новый слой поверх существующего direct/relay pipeline.
