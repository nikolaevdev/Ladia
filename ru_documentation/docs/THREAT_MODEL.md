# Модель угроз (краткая практическая версия)

## 1. Область охвата

Это краткая практическая модель угроз для Ladya в её текущем состоянии и в целевой архитектуре ближайшего Stage 5.
Документ намеренно сделан компактным и сфокусированным на тех угрозах, которые действительно важны для децентрализованного мессенджера в локальной сети.

## 2. Активы, которые нужно защищать

Основные активы:
- содержимое сообщений;
- файлы, voice notes и image payload;
- медиапотоки звонков;
- идентичность узла и trust-state;
- целостность маршрутизации;
- локальная история чатов;
- уверенность пользователя в том, что он общается именно с ожидаемым peer-узлом.

## 3. Типы противников

### A. Пассивный локальный наблюдатель
Возможности:
- видит локальный сетевой трафик;
- наблюдает поток пакетов;
- может записывать трафик для последующего анализа.

### B. Активный вредоносный peer
Возможности:
- участвует в discovery;
- может подключаться и отправлять crafted packets;
- может пытаться делать spoofing, spam, malformed data и route abuse.

### C. Вредоносный relay-узел
Возможности:
- пересылает трафик;
- видит packet metadata;
- может отбрасывать, задерживать, повторно воспроизводить или менять пакеты.

### D. Самозванец по идентичности
Возможности:
- выдаёт себя за другого peer;
- предъявляет изменённый key material;
- пытается использовать отсутствие верификации.

### E. Компрометация локального устройства
Возможности:
- злоумышленник получает доступ к телефону или хранилищу приложения.

## 4. Основные угрозы

### Угроза 1 — перехват трафика
Риск:
пассивный наблюдатель читает содержимое сообщений или файлов.

Текущее смягчение:
- уже есть задел по identity/authentication.

Требуемое смягчение на Stage 5:
- end-to-end AES-GCM шифрование payload;
- deriving session keys для каждой peer-пары.

### Угроза 2 — подмена peer-узла / key substitution
Риск:
вредоносный узел выдаёт себя за другого пользователя.

Текущее смягчение:
- подписанный handshake;
- отображение fingerprint;
- детекция suspicious key change;
- ручные состояния verify/block.

Дополнительное смягчение:
- identity keys на базе Android Keystore;
- QR-верификация fingerprint.

### Угроза 3 — чтение payload вредоносным relay-узлом
Риск:
промежуточный узел читает сообщения или содержимое файлов.

Текущее смягчение:
- концептуально relay должен видеть только transport metadata, но конфиденциальность payload ещё не обеспечена полностью.

Требуемое смягчение на Stage 5:
- relay пересылает только encrypted blob.

### Угроза 4 — модификация пакетов в пути
Риск:
атакующий меняет payload во время передачи.

Смягчение:
- authenticated encryption через AES-GCM;
- integrity checks для file containers / manifest;
- signature-based identity exchange.

### Угроза 5 — replay attacks
Риск:
старые пакеты повторно отправляются, чтобы запутать состояние системы.

Текущее смягчение:
- packet IDs;
- deduplication;
- TTL handling.

Дополнительное смягчение:
- timestamps и replay-check внутри secure container validation.

### Угроза 6 — злоупотребление routing / loop / congestion
Риск:
вредоносный или сломанный узел создаёт петли маршрутов, flooding или лишнюю пересылку.

Текущее смягчение:
- packet IDs;
- TTL;
- relay deduplication;
- reroute logic;
- diagnostics.

Дополнительное смягчение:
- более жёсткий rate limiting;
- throttling spam-трафика по peer.

### Угроза 7 — spam / noisy peers
Риск:
узел отправляет слишком много сообщений или ложных соединений.

Текущее смягчение:
- базовые trust controls;
- состояние blocked peer.

Рекомендуемое будущее смягчение:
- per-peer rate limits;
- soft ban;
- relay admission control.

### Угроза 8 — раскрытие локального хранилища
Риск:
если устройство скомпрометировано, история чатов может быть прочитана.

Текущее смягчение:
- локальное хранение есть, но это ещё не hardened secure vault.

Рекомендуемое будущее смягчение:
- минимизировать хранение чувствительного plaintext;
- по возможности шифровать выбранные локальные записи;
- использовать Android Keystore там, где это возможно.

### Угроза 9 — неправильное применение или ложная интерпретация стеганографии
Риск:
stego payload превышает вместимость carrier, повреждается или создаёт ложное ощущение безопасности.

Принцип смягчения:
- стеганография скрывает факт передачи, а не заменяет шифрование;
- поэтому она должна оборачивать уже зашифрованный secure container.

## 5. Какие security-гарантии стоит ставить целью

### От чего Ladya должна защищать
- от пассивного локального прослушивания;
- от подмены узла без верификации ключа;
- от чтения payload промежуточным relay;
- от ошибочного выбора недоверенных маршрутов;
- от незаметной модификации файлов и сообщений;
- от дублирования пакетов и усиления loop-сценариев.

### Чего Ladya не должна обещать сверх меры
- полной защиты при полностью скомпрометированном устройстве;
- гарантированной анонимности против любого traffic analysis;
- production-grade устойчивости против трафиковой корреляции уровня nation-state;
- идеальной секретности metadata в текущем MVP-формате.

## 6. Минимально приемлемый результат Stage 5

Разумная минимальная планка безопасности для хакатонной версии:
- identity keys в Keystore;
- верификация fingerprint;
- deriving session keys для peer-пары;
- AES-GCM-шифрование текстового payload;
- encrypted file containers;
- relay-узлы не могут расшифровать пересылаемый payload;
- видимая диагностика decryption failures и состояния secure-session.

## 7. Практический итог

Текущий репозиторий уже имеет хороший фундамент для:
- trust-aware discovery;
- видимости fingerprint;
- signed handshake;
- suspicious peer detection.

Stage 5 должен превратить этот фундамент в реальную конфиденциальность payload, чтобы система стала не только децентрализованной, но и действительно защищённой при работе в mesh-среде.
