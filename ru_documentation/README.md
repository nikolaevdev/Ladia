# Ladya
Перевод на русский, ничего нового с 21:00 нет.

**Ladya** — Android-прототип децентрализованного P2P-мессенджера без центрального сервера.
Приложение предназначено для автономной связи между устройствами внутри одной локальной Wi‑Fi-сети и уже поддерживает прямые сессии, relay-доставку, обмен файлами, голосовую связь и 1:1 видеосвязь.

Текущий снимок репозитория соответствует состоянию **Stage 4.8.5**.  
**Stage 5** находится в разработке и посвящён безопасности: end-to-end encryption, secure containers и стеганографическому транспорту.

## Цель проекта

Проект разрабатывается как децентрализованная система связи для сценариев, где централизованная инфраструктура отсутствует, нестабильна, перегружена или нежелательна. Целевые сценарии соответствуют требованиям хакатона: обнаружение узлов, установление P2P-сессий, обмен сообщениями, multihop relay, передача файлов, real-time-коммуникации, базовая аутентификация узлов, шифрование, диагностика и документированная архитектура.

### Уже реализовано

- локальный профиль узла;
- публикация и обнаружение узлов в локальной сети;

- MVP 1:1 видеозвонка по UDP;
- адаптация bitrate / облегчённый профиль видеопередачи;
- foreground service и уведомления.

#### Mesh- и групповой слой Stage 4
- self-healing восстановление после потери узла;
- relay-доставка текстовых сообщений;
- relay-доставка файлов;
- trust-aware выбор маршрута;
- rerouting через несколько соседей;
- mesh-диагностика и счётчики топологии;
- основа групповых чатов;
- приглашения в группы и управление составом;
- групповые текстовые сообщения через mesh;
- relay файлов в группах;
- доставка голосовых сообщений в группах;
- отдельный экран группы;
- настройки групп, проверка ролей, логика owner/admin;
- сохранение вложений по требованию пользователя прямо из bubble сообщения.

### Уже присутствует как задел по безопасности

Хотя Stage 5 ещё не завершён, в репозитории уже есть первые строительные блоки безопасности:
- генерация идентичности устройства и локальное хранение;
- вычисление fingerprint публичного ключа;
- подписанный HELLO / HELLO_ACK / HELLO_CONFIRM handshake;
- short authentication string для ручного сравнения идентичности;
- статусы доверия: `Unverified`, `Verified`, `Suspicious`, `Blocked`;
- stego trust capsule для встраивания подписанных trust-метаданных в PNG-изображения.

## Что добавит Stage 5

Stage 5 расширяет существующий транспорт, не ломая mesh-core:
- identity keys в Android Keystore;
- session keys для каждого peer через X25519 + HKDF;
- end-to-end encryption текстовых сообщений через AES-GCM;
- encrypted secure containers для файлов, голоса и изображений;
- стеганографический транспорт для скрытой доставки полезной нагрузки;
- новые diagnostics counters для encrypted/stego трафика;
- UX верификации идентичности через fingerprint и QR.

## Фактическая транспортная модель в этом репозитории

Данный снимок репозитория сейчас работает поверх **локального Wi‑Fi / LAN discovery**, а не через Интернет и не через центральный backend.
Реализованный сетевой стек выглядит так:

- **NSD (Network Service Discovery)** — публикация и обнаружение узлов в одной сети;
- **TCP socket** — основная прямая сессия обмена сообщениями;
- **TCP relay socket** — mesh-forwarding через промежуточные узлы;
- **TCP file-transfer socket** — доставка файловых payload;
- **UDP audio transport** — голосовые пакеты;
- **UDP video transport** — облегчённый 1:1 видеостриминг.

Иными словами, приложение уже децентрализовано, но текущую кодовую базу нужно понимать как **P2P / mesh-прототип для локальной сети**. Если в дальнейшем проект будет расширен до Wi‑Fi Direct или Bluetooth-based discovery, это будет архитектурное развитие, а не текущая runtime-модель.

## Архитектура верхнего уровня

Приложение организовано вокруг одного центрального уровня координации:

- `LadyaNodeRepository` — главный координатор состояния сети, сессий, UI, persistence, routing, relay, profile sync, file transfer, group logic и звонков.

Поддерживающие модули:
- `network/protocol/PacketEnvelope.kt` — транспортный envelope;
- `network/protocol/PacketFactory.kt` и `PacketParser.kt` — формирование и разбор пакетов;
- `network/protocol/TransferManifest.kt` — модель manifest передачи файлов;
- `call/AudioCallManager.kt` — UDP-аудио пайплайн;
- `call/VideoCallManager.kt` — UDP-видео пайплайн;
- `security/DeviceIdentityManager.kt` — локальная идентичность, fingerprint, подписи, SAS;
- `security/StegoTrustCapsuleManager.kt` — скрытые trust-capsule в изображениях на базе LSB;
- `LadyaDatabaseHelper.kt` — SQLite persistence;
- `ui/` — Compose-экраны, компоненты и UI-модели;
- `navigation/` — маршруты приложения и host навигации.

Более детальный разбор приведён в [`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md).

### Основные пользовательские сценарии

1. Пользователь запускает приложение.
2. Инициализируются foreground service и локальный repository.
3. Узел публикует себя в локальной сети и запускает discovery.
4. Пользователь видит найденные peer-устройства и может подключиться к ним.
5. После установления сессии пользователь может:
   - отправлять текстовые сообщения;
   - отправлять файлы;
   - записывать и отправлять голосовые сообщения;
   - запускать 1:1 аудиозвонок;
   - запускать 1:1 видеозвонок;
   - создавать группу;
   - управлять доверием и просматривать профиль.
6. Если прямой маршрут до целевого узла недоступен, relay и rerouting-логика могут доставить трафик через промежуточные peer-узлы.

## Модель безопасности в текущем снимке

Текущее состояние проекта следует рассматривать как **аутентифицированный транспорт с trust-контролями**, но **ещё не как полноценный end-to-end encrypted messenger**.

Уже реализовано:
- криптографическая идентичность устройства;
- подписанный handshake между узлами;
- сравнение fingerprint;
- детекция подозрительной смены ключа;
- состояния verified и blocked;
- trust-aware выбор маршрута.

Ещё не завершено:
- полное E2EE-шифрование payload для всех типов сообщений;
- deriving session keys для каждого peer через X25519;
- encrypted secure containers для файлов / голоса / изображений как транспорт по умолчанию;
- mesh-level обёртка encrypted payload для всех relay-данных.

## Как запустить проект

1. Клонируйте репозиторий.
2. Откройте его в Android Studio.
3. Убедитесь, что установлен Android SDK.
4. Откройте модуль `app` и выполните Gradle Sync.
5. Запустите приложение минимум на двух устройствах, подключённых к одной сети.
6. Разрешите доступ к микрофону, камере и уведомлениям, где это требуется.

## Рекомендуемые тестовые сценарии

### Базовая связность
- запустить приложение на двух устройствах;
- убедиться, что оба узла появились в discovery;
- установить прямую сессию;
- проверить двустороннюю доставку текстовых сообщений.

### Relay / mesh
- оставить одно устройство в роли relay;
- маршрутизировать через него текст или файл;
- проверить рост диагностических счётчиков;
- симулировать потерю маршрута и убедиться, что срабатывает reroute recovery.

### Передача файлов
- отправить небольшой файл;
- проверить manifest, progress и completion;
- убедиться, что получатель может сохранить и открыть файл.

### Аудио / видео
- установить звонок;
- проверить счётчики пакетов и длительность;
- протестировать mute микрофона / speaker mode / camera toggle.

### Групповой режим
- создать группу;
- отправить групповой текст;
- проверить owner/admin role checks;
- удалить участника;
- переименовать группу.

## Известные ограничения текущего снимка

- центральный repository очень крупный и совмещает много обязанностей;
- mesh-реализация остаётся MVP и пока не является полноценным распределённым routing-слоем;
- Stage 5 E2EE ещё в процессе;
- secure containers ещё не стали транспортом по умолчанию для всех типов сообщений/файлов;
- текущая стеганография ограничена trust-capsule в изображениях;
- сборка не была повторно провалидирована внутри этой среды, потому что Gradle wrapper не мог скачать distribution без внешнего сетевого доступа.
